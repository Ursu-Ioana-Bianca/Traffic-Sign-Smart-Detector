1. Internal Data Structures and Models

The web application maintains structured data for various entities such as users, traffic signs, reports, and notifications. The main data structures are as follows:

  • User Data Model: Consists of user profiles containing email, profile picture, preferences, and location history.
  • Traffic Sign Model: Stores sign identifiers, category, description, image, and geolocation (latitude and longitude).
  • Report Model: Includes reports submitted by users about traffic signs, their status, and associated timestamps.
  • Notification Model: Handles system-generated alerts and reminders for users.

2. API Implementation

The system employs a RESTful API for handling client-server interactions. Key API endpoints include:

  • GET /api/users/{email} - Retrieves user profile information.
  • POST /api/users/upload-profile - Updates user profile picture.
  • GET /api/traffic-signs/nearby - Fetches traffic signs based on user geolocation.
  • POST /api/reports - Allows users to report traffic sign issues.
  • GET /api/notifications - Retrieves notifications relevant to the user.

All requests are processed with appropriate authentication and error handling mechanisms to ensure secure data access.

3. RDF-Based Knowledge Models

The system leverages RDF-based knowledge representation to enhance the semantic interoperability of traffic sign data. The adopted ontology integrates:

  • Schema.org for representing structured metadata of traffic signs.
  • Wikidata properties to enrich traffic sign descriptions with globally recognized identifiers.
  • GeoSPARQL for handling spatial relationships of traffic signs.

4. External Knowledge Sources and SPARQL Queries

The application enriches its data by querying external knowledge bases. Key integrations include:

  • Wikidata Queries: Fetching additional metadata about traffic signs using SPARQL.
  • DBpedia: Linking relevant traffic regulations and concepts.
  • Linked Open Data: Ensuring interoperability with global traffic databases.

Example SPARQL query to retrieve traffic sign information:


        SELECT ?sign ?label WHERE {
            ?sign wdt:P31 wd:Q186103 .  # Traffic signs category
            ?sign rdfs:label ?label .
            FILTER (LANG(?label) = "en")
        }
        LIMIT 10

5. Linked Data Principles

The system adheres to linked data principles by:

  • Using dereferenceable URIs for traffic signs.
  • Integrating data from multiple RDF-based sources.
  • Enabling SPARQL endpoints for querying enriched traffic data.

Conclusion

The web application effectively manages traffic sign data while incorporating RESTful APIs and linked data principles. Future improvements include enhancing RDF schema expressiveness and optimizing API performance.
